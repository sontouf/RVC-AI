# Cursor 모드·도구 전환 플레이북 (Agent / Plan / Multitask)

목표: **문서 산출물**, **코드·테스트**, **실행·검증** 각각에 가장 적합한 Cursor 모드와 에이전트 사용 패턴을 고정합니다.

## 권장 매핑

| 활동 | Cursor 모드 | 이유 |
|------|-------------|------|
| 요구사항 ↔ 유스케이스 ↔ 품질속성 간 **모순 검토**, 단계 경계 합의 | **Plan** | 구현 권한 없이 구조·트레이드오프에 집중 |
| `arch/` 문서 갱신, Mermaid 다이어그램, 템플릿 채우기 | **Agent** | 파일 편집이 필요 |
| C++/CMake/GoogleTest 작성, 리팩터, 로컬 빌드·테스트 실행 | **Agent** | 코드·터미널 루프가 중심 |
| 시스템 테스트·커버리지·Actions 로그 분석 | **Agent** | 로그 기반 수정과 재실행 |
| **독립된 두 줄기**(예: 문서 톤 통일 vs CMake 정리) | **Multitask** | 단, **의존 관계가 있는 산출물**(예: 클래스 확정 전 대규모 구현)은 분리 금지 |

## 컨텍스트 유지 방법

에이전트는 채팅이 바뀌면 이전 합의를 잊기 쉽습니다. 다음을 **항상 같이** 넘깁니다.

1. 이 저장소의 Rules가 요구하는 **산출물 순서 요약** (`/.cursor/rules/`).
2. 직전 단계 산출물 **파일 경로 목록** (예: `arch/requirements/system-requirements.md`).
3. 현재 단계의 **완료 정의**(예: “유스케이스가 SYSREQ의 각 기능 요구와 추적표로 연결됨”).

## 멀티에이전트(병렬) 안전 구역

| 병렬 가능 | 병렬 비권장 |
|-----------|-------------|
| 문서 언어 교정 vs 코드 스타일 정리 | 도메인 모델 확정 vs 클래스 다이어그램 확정 |
| CI 워크플로 분리(job 단위 문서화) vs 테스트 케이스 추가 | 시스템 시퀀스와 시퀀스 다이어그램이 같은 유스케이스를 **서로 다른 계약**으로 가정할 때 |

## 모델 선택(실무 규칙)

- **설계·추적성·모순 탐지**: 상위(추론에 강한) 모델을 우선.
- **반복적인 테스트/리팩터**: 속도 위주 모델도 가능. 실패 시 로그를 붙여 상위 모델로 전환.
- **실행 결과가 소스 오브 트루스**: 모델 설명보다 `ctest`, `run_all.py`, Actions 로그를 우선합니다.
